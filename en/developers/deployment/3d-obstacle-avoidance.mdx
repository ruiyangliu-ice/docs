title: "Deploy 3D visual obstacle avoidance"
description: "Configure 3D visual obstacle-avoidance regions, extrinsics, and communication outputs with S Series and M Series cameras."
---

3D visual obstacle avoidance runs target detection, ground adaptation, and obstacle-status output on the camera. After configuring camera extrinsics and regions, you can switch obstacle-avoidance parameter groups and read results through the host application, API, or a supported field bus.

<Info>
  The values in this guide illustrate configuration relationships only. Actual values depend on the vehicle, installation position, and work area. Validate them on site before sending the configuration.
</Info>

## Prepare for deployment

### Camera installation

| Vehicle type | Installation guidance |
| --- | --- |
| Underride AMR | Mount the camera on the front surface in the direction of travel. The mounting structure must not obstruct the field of view. S10 is recommended. |
| Forklift | Mount the camera on the mast with the lens angled downward about `30°–55°`. The edge of the field of view should reach the front of the vehicle. Keep the installation height at or below `2 m` when possible. S10 is recommended. |

### Connect the camera

<Steps>
  <Step title="Configure the host network">
    The factory camera address is `192.168.100.82`. Configure the computer's Ethernet adapter on the same subnet.
  </Step>
  <Step title="Connect and verify imaging">
    Open **LxCameraViewer**, discover and open the camera, and confirm that the depth map and point cloud display correctly.
  </Step>
  <Step title="Change the camera address when needed">
    In **Other tools → Network → IP configuration tool**, change the IP address. The camera restarts after the change. If the camera moves to another subnet, update the computer's Ethernet address as well.
  </Step>
</Steps>

<Warning>
  Do not permanently disable the firewall on a production network. In a controlled debugging network, allow only the required applications and ports according to your security policy.
</Warning>

### Tune imaging

In **3D settings** and **Filter settings**, verify image quality. For legacy obstacle-avoidance configurations, a high-exposure value of `650`, low exposure of `200`, and low-signal threshold of `10–20` can be used as a starting point. S10 normally requires only a glare level of `1`. For non-standard installations, confirm the settings with technical support before deployment.

## Configure the obstacle-avoidance algorithm

<Steps>
  <Step title="Enable the algorithm">
    In **Application algorithm**, select the always-on mode. For S10, select **Obstacle avoidance algorithm**. For other supported models, select **Obstacle avoidance algorithm 2.0**.
  </Step>
  <Step title="Choose the region type">
    Choose a rectangular or sector-shaped region for the vehicle. S10 does not support sector-shaped regions; using one can produce incorrect results.
  </Step>
  <Step title="Set extrinsics and regions">
    Set the camera extrinsics first, then configure the obstacle-avoidance regions. When extrinsics change, the other regions use the new values.
  </Step>
  <Step title="Send and verify">
    Send the parameters, then verify warning, alarm, and shielded areas in the point-cloud view and output status.
  </Step>
</Steps>

### Camera extrinsics

| Parameter | Example | Description |
| --- | ---: | --- |
| Roll (X axis) | `180°` | Camera orientation; example for the default upright installation. |
| Pitch (Y axis) | `90°` | Camera orientation; example for the default vertical installation. |
| Yaw (Z axis) | `90°` | Camera orientation. |
| X position | `430 mm` | Camera position along the vehicle's front-to-rear axis. |
| Y position | `0 mm` | Camera position along the vehicle's left-to-right axis; `0` means centered. |
| Z position | `120 mm` | Camera height above the ground. |

Set the mounting height first, then fine-tune roll and pitch until the ground is level in the point-cloud view. Select **Send parameters** when finished.

### Obstacle-avoidance regions

| Parameter | Example | Purpose |
| --- | ---: | --- |
| Warning distance (far X) | `2000 mm` | Distance beyond which the area is considered clear; the interval to the alarm distance is the warning/deceleration zone. |
| Alarm distance (middle X) | `1500 mm` | The interval to the shield distance is the alarm/stop zone. |
| Shield distance (near X) | `600 mm` | Near-end blind zone; obstacles in this zone are excluded from obstacle avoidance. |
| Left / right range (Y) | `−810 / 1270 mm` | Detection limits on both sides of the vehicle centerline. |
| Lower / upper height (Z) | `10 / 1500 mm` | Height range included in detection. |

<Tip>
  X values are relative to the vehicle center. The actual protection distance must also account for the camera's X offset, such as warning distance minus the camera's front-to-rear offset.
</Tip>

The number of supported region groups depends on the switching method:

| Switching method | Supported groups |
| --- | ---: |
| S2 / S2 Max 2.0 I/O | `0–3` |
| SDK, UDP, RS485, or CAN | `0–19` (model dependent) |

Back up verified values in `Tools\\params\\<camera ID>\\avoidance.json`. You can apply the file to another vehicle of the same type with **Load configuration from file**.

## Communication and status output

Configure the obstacle-avoidance regions and extrinsics in the host application before using a field bus or network protocol. Except for the API, most communication methods switch parameter groups; use the host application or API for detailed parameter settings.

| Camera | Supported communication |
| --- | --- |
| S2 | API, UDP, I/O |
| S2 Max | API, UDP, CAN |
| S2 Max 2.0 | API, UDP, I/O, CAN |
| M4 Pro | API, UDP, RS485 |
| M4 | API, UDP, CAN |
| M4 Mega | API, UDP, I/O, CAN |

<Note>
  I/O has the highest priority. While I/O is active, other methods cannot change the current parameter group, but they can still update default or temporary parameters. The host application and API set default parameters; UDP, CAN, and RS485 normally set temporary parameters. Behavior after power loss depends on how the latest configuration was saved.
</Note>

### API

Windows, Linux, and ARM applications can call obstacle-avoidance APIs through the SDK. After installing the Windows host application, the SDK and examples are available in its installation directory. The obstacle-avoidance example is in `Sample\\C\\application_obstacled`. For S10, avoid switching parameter groups at a high frequency through the API; use UDP or I/O instead.

### RS485 (M4 Pro only)

Use Modbus RTU with a recommended baud rate of `9600`. Read the two input registers starting at address `108` (`0x6C`) with function code `0x04` to obtain the current obstacle-avoidance result. Write a single register at address `106` (`0x6A`) with function code `0x06` to switch parameter groups.

| Pin | Signal | Wire color |
| ---: | --- | --- |
| 1 | GND | Blue |
| 2 | RS485_A | Gray |
| 3 | RS485_B | White |
| 4 | VIN+ | Brown |
| 5 | VIN− | Black |

### UDP and TCP

Messages are sent as hexadecimal frames. For UDP, send the connection-establishment message first. After the connection is established, the camera sends results at the image frame rate.

| Direction | Frame | Meaning |
| --- | --- | --- |
| Camera → controller | `ACED0410XX000000` | No obstacle; `XX` is the current parameter group. |
| Camera → controller | `ACED0410XX000001` | Warning/deceleration. |
| Camera → controller | `ACED0410XX000002` | Alarm/stop. |
| Controller → camera (UDP) | `ACED040000000000` | Establish the UDP connection. |
| Controller → camera | `ACED041A00000000` to `ACED041A00000013` | Switch parameter group `0–19`. |
| Controller → camera | `ACED041B00000000` | Save the current group as the default group. |

The default remote UDP port is `6688`. Switching parameters requires load time; avoid rapid switching because it can reduce the image frame rate and result refresh rate.

### CAN and I/O

CAN can switch parameter groups on supported models. I/O outputs report clear, deceleration, and emergency-stop states. The output type may be push-pull or open-drain; confirm the controller input type before wiring.

| Push-pull OUT1 | Push-pull OUT2 | State |
| --- | --- | --- |
| `24 V` | `24 V` | No obstacle |
| `24 V` | `0 V` | Deceleration |
| `0 V` | `24 V` | Emergency stop |
| `0 V` | `0 V` | Disabled, initializing, or fault |

<Warning>
  For S10, connect `24 V−` to the controller power ground, and connect `IN_G` and `OUT_G` to the controller signal ground. If the controller does not distinguish the two grounds, use the site's approved common-ground scheme. Confirm whether the controller uses PNP or NPN signals and verify wiring with a multimeter.
</Warning>

## Troubleshooting

| Symptom | Checks |
| --- | --- |
| Black or low-reflectivity objects are missed | Confirm point-cloud coverage; adjust integration time and the low-signal threshold. Add a reflective surface to a black fork tip if needed. |
| Low objects do not trigger detection | Check the lower Z limit; temporarily set it to `0` for verification. If detection still fails, check mounting height and point-cloud coverage. |
| Parameters do not take effect | Confirm that they were sent. Check parameter validity, network connection, and parameter-group priority. |
| Abnormal data | Check for obstructions in the mounting structure, a lens protective film, or additional protective glass. |
