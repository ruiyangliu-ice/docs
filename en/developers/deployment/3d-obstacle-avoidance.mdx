## 1. Overview

Visual obstacle avoidance is based on Meyer Micro Vision's self-developed S series cameras and RGBD cameras based on the TOF principle. It integrates deep learning and traditional 3D vision algorithms to create an obstacle avoidance system that is suitable for rich scenes, accurate, real-time, single-machine configurable, multi-machine collaborative, and has a certain degree of initiative and predictability.

The system has the following characteristics: real-time dynamic detection, adaptive ground detection, dynamic tracking of detection targets, supports multiple signal outputs, supports multiple mode switching, can support 5 modes, and supports multi-machine collaborative work.

## 7. Communication method

1. Before understanding or using various communication methods, **you must use the host computer we provide to configure the contents of the obstacle avoidance parameters one by one** before you can use **non-API communication methods** to switch different obstacle avoidance parameters. **Note:**

**① Some communication protocols are unique to a certain model, such as RS485 communication is unique to M4Pro; **

② When using but not limited to the API method to obtain obstacle avoidance results, it is recommended to first confirm whether the obstacle avoidance configuration is switched successfully, that is, first query the current obstacle avoidance configuration area number;

③ Use UDP and CAN obstacle avoidance communication methods to check the obstacle avoidance area. For details, see the detailed introduction of each communication method.

| 2 | Others (such as 485, etc.) | The settings are temporary parameters. This parameter will be modified if there is a power outage or other communications. It can be saved as the default parameters through instructions, |

3. For modification of obstacle avoidance parameters, the camera will strictly follow the logic of priority.

4. The biggest difference between temporary parameters and default parameters is whether they are saved after power off. Which parameter is used depends on what communication method was used last time.

5. During IO communication (IO is not suspended), the current parameters cannot be modified using the communication method of priority 2, but the default parameters and temporary parameters can be modified. When IO communication is disconnected (IO is suspended), the specific default parameters or temporary parameters are used according to the last setting method. The last time 485 or UDP settings are used, temporary parameters are used. The last time API calls or host computer settings are used, the default parameters are used. The logic is as follows:

### 7.5 CAN communication method

1. CAN communication only supports S2-MAX cameras. The communication protocol uses CanOpen. The CanOpen protocol is an application layer protocol running on the standard CAN bus. Its communication mode is also the "master-slave" mode communication commonly used in industrial communication protocols. That is, there will be a master station + multiple slave stations in the network. The slave stations will not directly communicate with each other. All communications are between the master station and the slave stations. The master station is also called the "client" and the slave station is also called the "server".

2. The underlying communication method uses the CAN standard frame format, that is, the CAN ID is 11 bits (0x000~7FF) and the data is 8 bytes.

3. The usage convention of CAN ID: Divide the 11-bit ID into 4-bit function code and 7-bit node ID, and CAN ID is also called COB ID:

## 8. Frequently Asked Questions

### 8.1 Black objects

Due to the TOF principle, S series cameras are affected by the reflectivity of the black pallet. With a 5% reflectivity of the black pallet, the S2max camera can be used to achieve imaging within a range of 2m. During debugging, the imaging effect of the black pallet can also be optimized through the high integration time and low signal threshold of the host computer software. If there is a black fork arm in the obstacle avoidance scene, it is recommended that the user add a higher reflectivity to the fork tip to assist in obstacle avoidance, such as spraying a small section in front of the fork tip with a paint surface of other colors.

### 8.2 Low ground objects

Objects on the ground cannot avoid obstacles. In obstacle avoidance applications, we generally use the ground as the basis for external parameter configuration. If obstacle avoidance cannot be achieved when encountering low obstacles on the ground, you can refer to the following solutions:

- The point cloud image of the host computer shows that no obstacle is selected. You need to confirm whether the obstacle avoidance height (bottom) is a positive value. If it is a positive value, it is recommended to temporarily set it to 0 to determine whether the obstacle can be detected. If it still cannot be detected, it is recommended to adjust the camera installation height.

- The point cloud image of the host computer can select the obstacle object but the obstacle avoidance result output is 0. This phenomenon may be caused by the incorrect configuration of the camera installation height. It is recommended that the user adjust the camera installation height.

### 8.3 Obstacle avoidance parameters do not take effect

- Did not click Setup Complete when configuring parameters.

- Modification of illegal parameters caused the setting to fail.

- The camera connection is unstable and disconnected, causing parameter delivery to fail.

### 8.4 Installation structure interference

- During installation, there are structures around the camera that block the camera's field of view, resulting in abnormal camera data.

- During the structural design, the camera design interferes with the rear view, causing abnormal camera data.

- The protective film on the camera mirror has not been removed, resulting in abnormal camera data.

- Other protective glass is installed on the camera mirror, causing data anomalies.
