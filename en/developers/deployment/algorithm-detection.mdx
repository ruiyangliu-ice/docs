This guide covers deployment, calibration, configuration, and result integration for MRDVS human, storage-location, and volume-detection algorithms. The algorithm runs on the camera; the algorithm host application is used for configuration, upgrades, and result inspection.

<Info>
  Algorithm firmware and the algorithm host application are delivered per project. Before flashing algorithm firmware, confirm that the camera is running an MDS firmware version. Obtain an installation package that matches the project and camera model.
</Info>

## Supported scope

### System components

| Component | Function |
| --- | --- |
| 3D camera | Captures RGB images and point clouds and runs the detection algorithm on the camera. |
| Algorithm host application | Configures algorithms, calibrates extrinsics, displays results, and upgrades algorithm firmware. |
| Industrial PC or upper-layer system | Runs the host application or reads results through a network interface. |
| LiDAR | `mid360`, used for the original radar-based solution configuration. |

### Algorithms and cameras

| Algorithm | Supported camera |
| --- | --- |
| Human detection | M4 Pro |
| Storage-location detection | S10 Pro |
| Volume detection | S10 and S10 Pro |

For product information, see the [S10 3D camera product page](/en/products/s-series/s10-overview).

### Host application environment

| Item | Recommended configuration |
| --- | --- |
| Operating system | Windows 10 or later |
| CPU | Intel® Core™ i5 (7th generation) or later |
| Memory | 8 GB or more |
| GPU | Intel® UHD Graphics or a higher-performance integrated or discrete GPU |

## Hardware and network deployment

### Install the camera

Use a rigid bracket to prevent movement or vibration after installation. Human detection normally uses a horizontal installation or an installation with an angle of no more than 30°.

### Connect the devices

<Steps>
  <Step title="Connect power and the network">
    Power the camera with a regulated 24 V DC supply. Use a Cat6A or Cat7 cable to connect the camera to a gigabit switch or controller.
  </Step>
  <Step title="Deploy multiple cameras">
    Install each camera at the height and position specified in the CAD plan. Connect all cameras through a gigabit switch and verify power and link status for each one.
  </Step>
  <Step title="Prepare test targets">
    Place the people, pallets, or goods to be detected in the camera view before calibrating extrinsics and configuring regions.
  </Step>
</Steps>

### Configure the host network

Put the host computer and camera on the same subnet. The original example uses the following plan:

| Item | Example |
| --- | --- |
| Host IP address | `192.168.100.xxx` |
| Subnet mask | `255.255.255.0` |
| Default gateway | `192.168.100.1` |

<Steps>
  <Step title="Open adapter settings">
    In Windows, open **Control Panel → Network and Sharing Center → Change adapter settings**, then select the Ethernet adapter connected to the camera.
  </Step>
  <Step title="Configure IPv4">
    Open **Internet Protocol Version 4 (TCP/IPv4)**, select **Use the following IP address**, and enter an unused static address on the camera's subnet.
  </Step>
  <Step title="Confirm the connection">
    Save the settings, confirm a gigabit link, and use the host application to discover or open the camera.
  </Step>
</Steps>

## Install and upgrade software

### Install the host applications

1. Obtain the camera host application from the [software downloads](https://zcnt5g3o2989.feishu.cn/wiki/Q8IYwGiGaiIvIyktiavczymanOe?from=from_copylink "Software downloads") page.
2. Install the camera host application outside the system drive.
3. Copy the project-delivered algorithm host application files into the `Tools` folder under the camera host application directory. Replace files with the same name, then double-click the executable to start it.

<Frame>
  <img src="/images/developers/deployment-algorithm-detection-picture-2.avif" alt="Camera host application installation" />
</Frame>

To upgrade only the algorithm host application, replace its `.exe` file with the delivered version.

### Upgrade algorithm firmware

<Warning>
  The camera restarts during flashing. Do not disconnect power or the network or close the host application. The package must match the camera model, MDS firmware version, and project algorithm.
</Warning>

<Tabs>
  <Tab title="Camera host application">
    1. Start `LxCameraViewer.exe` and open the camera.
    2. Select **Basic tools → Function settings → Firmware upgrade**.
    3. Choose the delivered detection algorithm file and select **Run**.
  </Tab>
  <Tab title="Algorithm host application">
    1. Select the target camera in the device list.
    2. Select **Upgrade firmware**, choose the detection algorithm file, and open it.
    3. Wait for the camera to appear in the device list again.
  </Tab>
</Tabs>

When the camera appears again after reboot, flashing is complete. A green camera background in the algorithm host application means that the camera is connected and the algorithm is available.

## Algorithm host application

### Startup and status

Start `LxApplicationViewer.exe`. A green icon in the device list indicates a connected camera. Select a device to switch the current view between RGB and point-cloud modes. Standard users can view results; administrators can modify camera and algorithm parameters.

<Frame>
  <img src="/images/developers/deployment-algorithm-detection-administrator.avif" alt="Algorithm host application main window" />
</Frame>

### Basic tools

| Function | Description |
| --- | --- |
| User switching | Switch permission levels. Administrators can configure camera and algorithm parameters. |
| Ideal ground | Show or hide the reference horizontal plane. |
| Extrinsic ground | Show or hide the ground calculated from extrinsic calibration. |
| Display cache | Leave unchanged. |
| Display algorithm | Show the currently enabled algorithm. |
| Export saved images | Export saved images. |
| Historical playback | View RGB images and ToF data logs. |
| Switch application algorithm | Choose the algorithm to run, then select **Save and send**. |
| Display detection results | Show the output from the current algorithm. |

### Status bar and playback

| Function | Description |
| --- | --- |
| RGB | Show the RGB image. |
| Point cloud | Show the point cloud. |
| Stop algorithm | Stop the current algorithm. It does not restart automatically until the camera is powered on again. |
| Upgrade firmware | Open the algorithm firmware upgrade workflow. |
| Restart camera | Restart the selected camera. |

When the status changes, the software saves one RGB frame and point-cloud data. Playback data is not shared between algorithms. Playback time uses the camera's internal clock, not the host computer's system time.

If the device background is yellow, or it is green but the image does not update, enable automatic refresh to resume image acquisition.

## Deploy an algorithm

Switch to point-cloud mode and scroll over the image area to display point-cloud data.

### Human detection

<Steps>
  <Step title="Calibrate a horizontal installation">
    Align the `X` and `Y` axes with the ground point cloud. The `Z` axis must be perpendicular to and point up from the ground.
  </Step>
  <Step title="Set global parameters">
    Open **Human detection**. Usually only **Rotation detection** needs to be set. Clockwise camera rotation supports `0°`, `90°`, `180°`, and `270°`.
  </Step>
  <Step title="Create the detection region">
    Select **Add detection region**, set minimum and maximum `X`, `Y`, and `Z`, and move the region over the target. The original example uses a `1000 × 1000 × 2000 mm` region.
  </Step>
  <Step title="Save and verify">
    Select **Save and send**, run a person-through-area test, and verify the result in the status bar or historical playback.
  </Step>
</Steps>

| Matching rule | No person | Person | Obstacle | Person and obstacle |
| --- | --- | --- | --- | --- |
| Depth only | `0` | `2` | `2` | `2` |
| Semantic plus depth | `0` | `1` | `0` | `1` |

Depth-only mode reports every object in the region. Semantic-plus-depth mode reports only clusters classified as people.

### Storage-location detection

<Steps>
  <Step title="Calibrate the camera">
    Follow the extrinsic-calibration procedure. Align `X` and `Y` with the ground and keep `Z` perpendicular and pointing up.
  </Step>
  <Step title="Configure global parameters">
    Set **Alignment check**, **Alignment angle**, and **Minimum point count** for the site. Enable alignment checking when cargo must be placed at a defined angle.
  </Step>
  <Step title="Create storage locations">
    In point-cloud mode, select **Add storage location**, switch to the top view, and press `X` to start or stop drawing. Drag the rectangle over the pallet and adjust the `X`, `Y`, and `Z` boundaries. Press `X` again to exit drawing mode.
  </Step>
  <Step title="Save and verify">
    Select **Save and send**. Configure locations in 2D RGB or 3D point-cloud mode and check `Result` in the status bar.
  </Step>
</Steps>

<Frame>
  <img src="/images/developers/deployment-algorithm-detection-point-cloud-mode-location-setting-4.avif" alt="Drawing a storage-location region in point-cloud mode" />
</Frame>

| Storage-location state | `Result` |
| --- | --- |
| Empty | `0` |
| Occupied, or occupied and aligned | `1` |
| Occupied with an angle beyond the alignment threshold | `2` |
| Occupied and outside the location boundary | `3` |

For a regular layout, use **Copy current box** to create locations at a fixed direction and spacing. If QR-code verification is required, use **QR Code calibration** to configure location dimensions and the depth range.

### Volume detection

<Steps>
  <Step title="Calibrate a vertical installation">
    Volume detection works from top to bottom. After calibration, the `Z` axis points down and the `X`/`Y` plane is parallel to the ground.
  </Step>
  <Step title="Set the detection plane">
    For goods placed directly on the floor, set the detection plane to the floor height. To measure only goods on a pallet, align the plane with the pallet point cloud.
  </Step>
  <Step title="Set minimum dimensions and the region">
    Set minimum `X`, `Y`, and `Z` values to filter small targets. Then set the region boundaries according to the detection plane and cargo dimensions.
  </Step>
  <Step title="Save and verify">
    Select **Save and send**, place one or more objects in the region, and verify the returned length, width, height, and count.
  </Step>
</Steps>

## Communication interfaces

<Note>
  The IP address in the examples is illustrative. Replace it with the camera address at the site, then verify the network, algorithm state, and result format before integrating with an upper-layer system.
</Note>

### Transport methods

| Protocol | Port | Request | Use |
| --- | --- | --- | --- |
| TCP | `14951` | `RESULT_UPDATE` | Get the current algorithm result. |
| UDP | `14950` | `RESULT_UPDATE` | Get the current algorithm result. |
| HTTP | `14952` | `RESULT_UPDATE` | Get the algorithm result as JSON. |
| TCP / UDP | Result port | `TM_TRIGGER` | Run one triggered volume detection and return the result. |
| Modbus TCP | `16502` | Read holding registers | Read storage-location status. |

### HTTP result request

The upper-layer system sends a result request to the vision endpoint, which returns a JSON string. Use the project-delivered request headers in your HTTP client and the following request body.

<ParamField body="command" type="string" required default="RESULT_UPDATE">
  Triggers the camera to return the current algorithm result.
</ParamField>

### Common response fields

<ResponseField name="IP" type="string" required>
  IP address of the camera that produced the data.
</ResponseField>

<ResponseField name="nick_name" type="string" required>
  Camera name; change it in camera configuration.
</ResponseField>

<ResponseField name="ret_code" type="integer" required>
  Return code; `0` indicates success.
</ResponseField>

<Tabs>
  <Tab title="Human detection (SG)">
    <ResponseField name="SG" type="object" required>
      Status object keyed by detection-region ID: `0` means no target, `1` means a person was detected, and `2` means an object was detected only by the depth rule.
    </ResponseField>
    <ResponseField name="__object_details" type="string[]">
      Cluster details. Each item is a JSON string that must be deserialized again.
    </ResponseField>
    <ResponseField name="SG_DATA_TIME" type="string">Human-detection data timestamp.</ResponseField>
    <ResponseField name="SG_RES_TIME" type="string">Human-detection result timestamp.</ResponseField>

    Cluster details include `rbx2d`, `rby2d`, `tlx2d`, and `tly2d` (2D box coordinates), `semantic_index`, `traceID`, and `x_min`, `x_max`, `y_min`, `y_max`, `z_min`, and `z_max` (3D box boundaries).
  </Tab>
  <Tab title="Storage-location detection (SS)">
    <ResponseField name="SS" type="object" required>
      Status object keyed by location ID: `0` empty, `1` occupied, `2` occupied with an angle offset, and `3` occupied and out of bounds.
    </ResponseField>
    <ResponseField name="X__detail__info" type="number[]">
      Details for location `X`: center `x`, `y`, and `z`, length, width, height, and angle offset from the bounding box. A location created in 2D returns all `-1` values.
    </ResponseField>
    <ResponseField name="SS_DATA_TIME" type="string">Storage-location data timestamp.</ResponseField>
    <ResponseField name="SS_RES_TIME" type="string">Storage-location result timestamp.</ResponseField>
  </Tab>
  <Tab title="Volume detection (TM)">
    <ResponseField name="TM" type="object[]" required>Array of complete volume boxes in the detection region.</ResponseField>
    <ResponseField name="angle" type="number">Volume-box angle offset.</ResponseField>
    <ResponseField name="cx" type="number">Volume-box center `x` coordinate.</ResponseField>
    <ResponseField name="cy" type="number">Volume-box center `y` coordinate.</ResponseField>
    <ResponseField name="cz" type="number">Volume-box center `z` coordinate.</ResponseField>
    <ResponseField name="length" type="number">Object length.</ResponseField>
    <ResponseField name="width" type="number">Object width.</ResponseField>
    <ResponseField name="height" type="number">Object height.</ResponseField>
    <ResponseField name="type" type="string">Semantic type.</ResponseField>
    <ResponseField name="valid" type="boolean">Whether the result is valid.</ResponseField>
    <ResponseField name="TM_DATA_TIME" type="string">Volume-detection data timestamp.</ResponseField>
    <ResponseField name="TM_RES_TIME" type="string">Volume-detection result timestamp.</ResponseField>
  </Tab>

### Modbus TCP (storage-location detection)

Results are ordered by storage-location ID; at most the first 16 locations are returned.

| Configuration | Value |
| --- | --- |
| Port | `16502` |
| Listen address | `0.0.0.0` (all network interfaces) |
| Register count | `16` |
| Start address | `0x0100` |
| Byte order | Big-endian |

| Register | Content | Data format |
| --- | --- | --- |
| `0x0100 + ID` | Storage-location and marker detection results | 2-byte `uint16_t` |
| High 8 bits | Storage-location status code | `0–2` |
| Low 8 bits | Marker status code | `0–2`; this feature is not currently available, so it has the same meaning as the location status code. |

| Location status | Meaning | Condition |
| --- | --- | --- |
| `0` | Empty | No goods or pallet. |
| `1` | Occupied normally | Goods are placed as required. |
| `2` | Occupied abnormally | Cargo size or position is outside the requirement. |

| Marker status | Meaning |
| --- | --- |
| `0` | Marker not detected. |
| `1` | Marker detected. |
| `2` | Marker detection error. |

#### Read the fourth storage location

The following example uses transaction ID `0x0001` to read one holding register at address `0x0104`. The response contains status `0x0002`.

```text
Request: 00 01 00 00 00 06 FF 03 01 04 00 01
Response: 00 01 00 00 00 05 FF 03 02 00 02
```

| Field | Request value | Description |
| --- | --- | --- |
| Transaction identifier | `00 01` | Client sequence number; the response must match it. |
| Protocol identifier | `00 00` | Fixed Modbus/TCP value. |
| Length | `00 06` | Number of bytes in the following request fields. |
| Unit identifier | `FF` | Broadcast mode or device address. |
| Function code | `03` | Read holding registers. |
| Start address | `01 04` | `0x0100 + 4`. |
| Register count | `00 01` | Read one register. |

## Troubleshooting

| Symptom | Priority checks |
| --- | --- |
| Camera not found | Confirm that the host and camera share a subnet, the network adapter is gigabit, and power and switch links are healthy. |
| Parameters do not take effect | Confirm that **Save and send** was selected after the change. |
| No algorithm result | Confirm that the correct algorithm firmware was flashed, the device is green, and the algorithm is enabled. |
| Empty or stale point cloud | Scroll over the point-cloud area to display data; enable automatic refresh if needed. |
| Abnormal location or volume results | Recheck extrinsic-axis directions, detection plane, and region boundaries, then retest with site objects. |
