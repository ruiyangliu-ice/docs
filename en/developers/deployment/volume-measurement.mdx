The volume-measurement algorithm runs on the camera and measures the length, width, and height of static or moving objects. Deployment requires a rigid installation, network connection, extrinsic calibration, and detection-region configuration. Connect multiple cameras to the same network through a gigabit switch.

<Info>
  This is a hidden historical deployment guide. Algorithm firmware and the dedicated host application are delivered per project. Confirm that the camera, firmware, and software versions match before deployment.
</Info>

## Scope

This workflow applies to logistics, parcel, warehousing, and bulk-material volume-measurement projects using M Series cameras. The algorithm measures one or more objects in the detection region. Object identification, weight collection, and business-system association are handled by the upper-layer system.

See the [M Series ToF cameras](/en/products/m-series/m4-overview "M Series ToF cameras") for product information.

## Installation and system preparation

### Hardware installation

- Use a rigid bracket to mount the M4 camera with the face parallel to and pointing toward the detection plane.
- Power the camera with `24 VDC`. Use a Cat6A or Cat7 cable to connect the camera to the host or a gigabit switch.
- For multiple cameras, assign each camera a unique network address and verify its power state.

### Host requirements

| Item | Recommended configuration |
| --- | --- |
| Operating system | Windows 10 or later |
| CPU | Intel® Core™ i5 (7th generation) or later |
| Memory | 8 GB or more |
| GPU | Intel® UHD Graphics or a higher-performance integrated or discrete GPU |

First install [LxCameraViewer](/en/developers/software/lxcameraviewer "Software download"), then place the project-delivered volume-measurement application in the `Lanxin-MRDVS\\Tools` directory. Replace files when prompted by the delivery instructions.

<Warning>
  The camera restarts during firmware upgrades. Use only the project-delivered algorithm package and back up verified configuration before upgrading.
</Warning>

## Deployment workflow

### 1. Configure the network

Put the host and camera on the same subnet. The original deployment example uses:

| Item | Example |
| --- | --- |
| Host IP | `192.168.100.xxx` |
| Subnet mask | `255.255.255.0` |
| Default gateway | `192.168.100.1` |

In the Windows Ethernet adapter properties, select **Internet Protocol Version 4 (TCP/IPv4)** and set a static address. Allow the application through the debugging network's firewall rules; do not permanently disable a production firewall.

### 2. Calibrate extrinsics

<Steps>
  <Step title="Restore the calibration state">
    Open **Camera configuration**. Confirm that `x`, `y`, `z`, `roll`, `pitch`, and `yaw` are `0`. If historical values exist, reset them and select **Save and send**.
  </Step>
  <Step title="Run calibration">
    Select **Extrinsic calibration**, confirm the success message, and record the resulting `z` value.
  </Step>
  <Step title="Save the pose parameters">
    Set `z` back to `0` and select **Save and send**. In this workflow, the actual camera height is used as the detection-plane distance in the next step.
  </Step>
</Steps>

### 3. Set the detection plane and region

In **Volume detection**, enter the recorded `z` value as **Detection-plane distance**, then save and send. Configure the detection-region boundaries from the point-cloud view:

| Parameter | Meaning |
| --- | --- |
| Minimum / maximum X | X boundaries of the detection box |
| Minimum / maximum Y | Y boundaries of the detection box |
| Minimum / maximum Z | Z boundaries of the detection box |

The detection-plane distance defines the floor reference; set the Z upper limit slightly above this distance. The lower limit must cover the object heights to measure. Select **Save and send** after configuration.

### 4. Verify results

View volume status in **Basic tools**. Place an object in the detection region and confirm that its count and length, width, and height match the site. For multiple objects, switch between volume boxes to inspect each result.

For single-shot measurement, enable **Trigger mode** in **Volume detection**, save the configuration, and select **Single trigger** in **Basic tools** to acquire one frame and result.

## Key interface and parameters

### Volume-detection parameters

| Parameter | Description |
| --- | --- |
| Trigger mode | When enabled, the algorithm outputs results only after a trigger. |
| Upward detection | Reverses the detection direction; the default workflow detects downward. |
| Detection frequency | Image and result refresh period. |
| Detection-plane distance | Distance from the camera to the detection plane, in `mm`. |
| Neighborhood count / distance threshold | Point-cloud filtering parameters; keep the project defaults in most cases. |
| Normal-vector filter / distance threshold | Edge and point-cloud filtering parameters; preserve the original configuration before changing them. |
| Minimum point count | Minimum number of points required for a valid object. |
| Minimum X / Y / Z | Minimum detectable object dimensions, in `mm`. |

### Camera configuration

| Item | Description |
| --- | --- |
| Camera name | Distinguishes cameras in a multi-camera deployment. |
| Image-save mode | Saves algorithm images on the camera. |
| Image-acquisition mode | `MDS` does not occupy camera access. `SDK` does; the host application cannot open the camera while it is running. |
| Extrinsics | `x`, `y`, `z`, `roll`, `pitch`, and `yaw`. |
| Save and send | Writes the current extrinsics and algorithm configuration to the camera. |

## Result interface

TCP, UDP, and HTTP use the `RESULT_UPDATE` command to request volume results.

| Protocol | Port | Use |
| --- | ---: | --- |
| TCP | `14951` | Request a volume-detection result |
| UDP | `14950` | Request a volume-detection result |
| HTTP | `14952` | Request the volume result as JSON |

<ParamField body="command" type="string" required default="RESULT_UPDATE">
  Requests the current volume-detection result from the camera.
</ParamField>

Top-level HTTP response fields:

<ResponseField name="IP" type="string" required>IP address of the camera that produced the data.</ResponseField>
<ResponseField name="TM" type="object[]" required>List of detected objects.</ResponseField>
<ResponseField name="TM_DATA_TIME" type="string">Time when volume data was captured.</ResponseField>
<ResponseField name="TM_RES_TIME" type="string">Time when the result was output.</ResponseField>
<ResponseField name="nick_name" type="string">Camera name.</ResponseField>
<ResponseField name="ret_code" type="integer" required>Return code; `0` indicates success.</ResponseField>

Each `TM` object can contain:

<ResponseField name="angle" type="number">Object-box angle.</ResponseField>
<ResponseField name="cx" type="number">Object-center X coordinate.</ResponseField>
<ResponseField name="cy" type="number">Object-center Y coordinate.</ResponseField>
<ResponseField name="cz" type="number">Object-center Z coordinate.</ResponseField>
<ResponseField name="height" type="number">Object height.</ResponseField>
<ResponseField name="length" type="number">Object length.</ResponseField>
<ResponseField name="width" type="number">Object width.</ResponseField>
<ResponseField name="valid" type="boolean">Whether the result is valid. `false` indicates insufficient confidence.</ResponseField>

## Troubleshooting

| Symptom | Recommended action |
| --- | --- |
| Point-cloud view is empty or hard to inspect | Scroll and drag in the point-cloud area and confirm that the camera outputs data. |
| Detection is abnormal after calibration | Confirm that extrinsic `z` was reset to `0` and that the recorded calibration height was entered as the detection-plane distance. |
| Objects are not detected | Check region boundaries, minimum dimensions, minimum point count, and whether the object is fully in view. |
| LxCameraViewer cannot open | Check whether `SDK` image-acquisition mode is selected; it occupies camera access. |
